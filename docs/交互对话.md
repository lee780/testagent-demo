## 对话一：测试用例不确定性问题

### 背景
用户发现同样的输入，每次生成的测试用例内容不同，希望了解原因并得到专业建议。

### 根因分析

**核心原因：LLM 本质上是随机的**

| 原因 | 说明 |
|------|------|
| temperature 未固定 | 主 Agent 无显式 temperature，使用模型默认值（0.7~1.0）；标题生成为 1.2 |
| 无 seed 参数 | 随机数种子未固定，GPU 浮点计算存在不确定性 |
| Prompt 约束不足 | System Prompt 只规定格式，未限定生成哪些具体用例，AI 有"创作自由度" |
| Agent 工具调用顺序不固定 | pi-mono Agent 框架中工具调用顺序本身是 AI 决策的，引入额外不确定性 |

### 关键洞察
> "测试用例的**设计决策**不应该依赖 LLM 的每次发挥，而应该由**确定性的等价类划分算法**驱动，LLM 只负责辅助理解文档和生成自然语言描述。"

### 建议措施

| 层次 | 措施 |
|------|------|
| 短期 | 设置 temperature=0、seed=42，降低随机性 |
| 中期 | 将用例生成决策从 LLM 剥离，用算法（BVA/EP）确定"生成哪些用例" |
| 长期 | 建立基线 YAML 文件，提交 Git，版本锁定，变更需审批 |

## 对话二：多种测试模式设计

### 背景
通过对不确定性问题的分析，用户意识到"探索性测试"和"固化测试"是两种本质不同的需求，希望平台支持多种模式。

### 模式框架

```
确定性 ◄──────────────────────────────────► 探索性
  │                                            │
回归模式   系统化模式   探索模式   混沌/对比模式
（复现）  （BVA算法）  （LLM发散）  （双系统差异）
```

### 最终确定四种模式

| 模式 | 标识 | 核心特征 | 适用场景 |
|------|------|---------|---------|
| 回归模式 | 🔒 | 100% 确定性，只跑已有基线 YAML | 版本发布前回归、CI/CD |
| 系统化模式 | 📐 | BVA + 等价类算法驱动，禁止随机值 | 新业务规则首次建设用例库 |
| 探索模式 | 🔭 | LLM 假设驱动，挖掘非常规缺陷 | 发现未知缺陷，对抗性测试 |
| 混沌/对比模式 | 🌪️ | 同一用例集跑两套系统，自动差异分析 | 重构验证、新旧系统对比 |

### 关键设计决策

- **回归模式**：找不到基线 YAML 时主动停止并提示用户"请先用系统化/探索模式建基线"
- **系统化模式**：用例 ID 必须顺序编号（TC_MODEL001_S01...），禁止 UUID 或随机字符串
- **探索模式**：每跑一个用例就分析结果，再决定下一步，而非批量生成后一次执行
- **混沌模式**：Hard Constraint——两次系统跑必须使用完全相同的 YAML 文件，禁止中间修改

## 对话三：四种模式实现

### 实现范围
全链路 6 个文件同步修改，mode 参数从前端 UI 到最终 System Prompt 完整串联：

```
前端 mode 选择器（ChatView.vue）
  → fetch body: { mode }
  → chat.schemas.ts: z.enum(['regression','systematic','exploratory','chaos'])
  → chat.routes.ts: body.mode → sendMessageStreaming
  → chat.service.ts: mode → runTestAgent
  → runner.ts: mode → buildSystemPrompt
  → system-prompt.ts: 4 套独立 Prompt
```

### 前端 UI 设计
- 4 个模式按钮嵌入输入框底部 action bar
- 各模式有独立颜色：蓝/绿/琥珀/红
- 默认选中系统化模式（📐）
- 空状态和有消息状态两处输入框保持同步

### 关键发现：P0 阻断缺陷（已修复）

**BUG-01**：服务器进程（01:07 AM 启动）运行旧版 schema（`standard|explore` 枚举），而代码在 08:10 AM 更新为四模式版本，`tsx` 不支持热重载，导致所有新 mode 值被 schema 拒绝。

**修复**：重启服务器进程，重新加载最新代码。

**验证结果**：
```

mode=systematic → HTTP 200 SSE: event: start + thinking ✓
mode=exploratory → HTTP 200 SSE: event: start ✓
mode=chaos      → HTTP 200 SSE: event: start ✓
mode=hacked     → 400 VALIDATION_ERROR: Expected 'regression'|'systematic'|'exploratory'|'chaos' ✓
```

## 对话五：文件下载面板重设计

### 问题
文件下载面板使用 `position: fixed` 悬浮在页面右上角，遮挡内容，无法收起，体验差。

### 方案
改为嵌入式折叠面板，位于消息区底部、输入框上方。

| 维度 | 改前 | 改后 |
|------|------|------|
| 定位 | fixed，悬浮右上角 | 嵌入页面流，在输入框上方 |
| 收起 | 不支持 | 点击标题栏展开/收起，带动画 |
| 宽度 | 固定 240px | 跟随聊天区宽度自适应（800~1400px） |
| 视觉 | 与页面割裂 | 顶部分隔线，与聊天区自然衔接 |
| 无文件时 | 不显示（不占位） | 不显示（不占位）—— 保持 |

### 实现
- `FileDownloadPanel.vue`：移除 `position: fixed`，改为 `border-top` 分隔，加 `expanded` ref 控制折叠，CSS `max-height` 动画过渡
- `ChatView.vue`：将面板从 `chat-messages` 内部移到 `bottom-input-wrapper` 上方

## 总结：关键技术决策

| 决策 | 内容 |
|------|------|
| 用例确定性 | 生成决策由算法驱动，LLM 只做文档理解和文字描述 |
| 多模式架构 | 4 种模式对应不同确定性级别，System Prompt 完全分离 |
| 默认模式 | 系统化模式（BVA算法）作为默认，覆盖最常见需求 |
| 测试驱动 | 功能实现后立即进行多维度验证，发现并修复 P0 缺陷 |
| UI 设计原则 | 嵌入式优于悬浮式，功能控件贴近使用场景 |

## 对话二：意外还原与数据丢失事故

### 经过
用户要求「还原到上次对话前的状态」，执行 `git restore .` 后，不仅撤销了本次改动，也把对话开始前已有的未提交修改一并永久丢失。

**永久丢失的文件改动（经多方尝试，无法恢复）：**
- `frontend/src/views/ChatView.vue`
- `src/agent-core/tools/test-runner.ts`
- `.claude/settings.local.json`
- `docs/01-需求讨论.md`、`02-产品设计文档.md`、`04-概要设计文档.md`
- `docs/05-详细设计文档.md`、`06-测试验收报告.md`、`08-测试架构组网图.md`
- `docs/testfile/MODEL001_业务规则文档.md`、`docs/产品介绍.md`

**尝试恢复的手段（均失败）：**

| 手段 | 结果 |
|------|------|
| `git stash list` | 无 stash |
| `git reflog` | 未提交内容不在 reflog，无帮助 |
| VSCode Local History | 只有 1 条不相关记录（`cline_mcp_settings.json`） |
| macOS APFS 本地快照 `tmutil listlocalsnapshotdates` | 无快照 |

**教训：**
> 未提交的本地修改在 `git restore .` 后不可逆。重要改动应及时 commit，哪怕是 `WIP` commit。
