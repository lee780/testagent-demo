## 一、产品概述

### 1.1 背景

传统接口测试流程依赖人工阅读业务规范文档、手工编写测试用例、逐条调用接口验证，效率低且容易遗漏边界场景。

TestPilot 在 AI 驱动的测试执行能力基础上，构建了完整的**测试管理闭环**：测试用例库（含生命周期管理和版本控制）+ AI 回归推荐，让高质量基线用例持续发挥价值。

### 1.2 产品定位

面向测试工程师、审核人和技术负责人，提供"AI 赋能的一站式接口测试管理平台"，将测试设计、执行、用例管理整合在一个平台中。

### 1.3 核心价值

| 价值点 | 说明 |
|--------|------|
| AI 驱动测试执行 | 自然语言对话触发测试，Agent 自动生成用例、执行、出报告 |
| 用例库生命周期管理 | DRAFT→BASELINE 全流程，确保基线用例经过人工审核 |
| AI 回归推荐 | 输入变更描述，自动推荐最相关的基线用例，提高回归效率 |

## 三、用户故事

### US-001：AI 对话驱动测试执行（核心）
**作为**测试工程师，
**我希望**通过自然语言描述测试需求，Agent 自动生成测试用例、执行接口测试并输出报告，
**以便**将测试准备时间从数天压缩到分钟级，同时保证边界场景覆盖率。

**四种测试模式说明**：

| 模式 | 实现状态 | 说明 |
|------|---------|------|
| 系统化 | ✅ 已实现 | BVA + 等价类驱动，用例 ID 顺序编号，适合首次建库 |
| 探索 | ✅ 已实现 | LLM 假设驱动，逐步分析挖掘非常规边界缺陷 |
| 回归 | ⚠️ 部分实现 | 现阶段以回归导向提示词重新执行；从数据库读取基线 YAML 精确回放为后续迭代目标 |
| 混沌 | ⚠️ 环境依赖 | 需同时运行两套系统（:8000 / :8001）进行差异对比，依赖双系统环境 |

**验收标准**：
- [ ] 输入框底部有四个模式按钮（系统化/回归/探索/混沌）
- [ ] 模式参数随消息发送，全链路传递到 System Prompt
- [ ] 测试执行过程中，聊天界面实时展示测试进度事件
- [ ] 执行完成后，聊天界面推送"查看报告详情"可点击卡片（不强制跳转，保留用户主动权）
- [ ] 报告中包含每条用例的入参、预期结果、实际结果及执行日志，内容多时折叠展示

### US-002：测试用例手动确认入库
**作为**测试工程师，
**我希望**在执行完成并查看报告后，由我手动选择是否将本次用例入库，
**以便**避免自动入库产生大量重复或低质量草稿，保持用例库整洁。

**验收标准**：
- [ ] 执行完成后，界面提供"确认入库"操作，不自动入库
- [ ] 用户确认后，用例以"草稿"状态保存入库
- [ ] 用例包含：模型编号、用例编码、标题、入参、预期结果、用例内容、来源模式、对话编号
- [ ] 入库后走原有审核流程（草稿 → 待审核 → 已审批 → 基线）

### US-003：测试用例生命周期管理
**作为**审核人，
**我希望**通过用例库界面对"待审核"状态的用例进行审批（批准或拒绝），
**以便**确保只有经过人工确认的高质量用例被晋级为"已审批"，再由技术负责人晋级为"基线"。

**验收标准**：
- [ ] "草稿"用例可被提交审核（状态变为"待审核"）
- [ ] 审核人可批准（→ 已审批）或拒绝（→ 草稿，含拒绝意见）
- [ ] "已审批"用例可被晋级为"基线"
- [ ] "基线"用例不可直接修改，修改需创建新版本
- [ ] 废弃操作将状态置为"已废弃"

### US-004：版本控制
**作为**测试工程师，
**我希望**在修改已有用例时，系统自动创建新版本而不是覆盖旧版本，
**以便**我可以回溯历史版本，了解用例的演进轨迹。

**验收标准**：
- [ ] 修改"已审批"或"基线"用例时，系统创建新版本记录，旧版本保留
- [ ] 用例详情页可查看版本历史列表
- [ ] 系统始终将最新版本作为当前有效版本

### US-005：AI 回归推荐与快捷执行
**作为**测试工程师，
**我希望**输入本次变更描述后，系统推荐最相关的"基线"用例，并支持对推荐结果一键触发回归执行，自动生成回归验证报告，
**以便**从变更描述到回归结论形成完整闭环，不遗漏高风险场景。

**背景说明**：
系统在数据库中维护结构化关联关系（用例 ↔ 文档、用例 ↔ 报告、用例 ↔ 对话），无需向量化存储。推荐时从数据库查询"基线"用例及其关联上下文，整体交给大模型直接推理排序。推荐结果可直接驱动执行，回归报告自动继承原有关联关系。

**完整链路**：
```
输入变更描述
    ↓ AI 推荐（LLM 语义推理 + 关联上下文）
推荐结果列表（含推荐理由，可追溯依据文档/对话）
    ↓ 用户勾选用例，点击「一键执行」
回归执行（复用基线用例的 YAML 内容）
    ↓
生成回归验证报告（标注"回归类型"，与首次执行报告有所区分）
报告自动关联：原始用例 + 对话 + 上传文档
```

**验收标准**：
- [ ] 用例入库时，系统自动记录关联关系（用例 ↔ 报告、用例 ↔ 对话、用例 ↔ 上传文档）
- [ ] 提供变更描述输入入口，提交后 LLM 结合关联上下文推理排序
- [ ] 推荐结果按相关度排序，每条附带推荐理由（可追溯关联文档或对话）
- [ ] 推荐结果页支持勾选用例，点击「一键执行」触发回归运行
- [ ] 回归执行完成后生成报告，报告类型标注为"回归验证"，与首次执行报告在列表中有视觉区分
- [ ] 回归报告自动关联原始用例、来源对话及上传文档
- [ ] 候选用例较多时，先按模型编号、来源模式粗过滤，再交大模型精排

### US-006：测试报告管理
**作为**测试工程师，
**我希望**有一个独立的测试报告管理模块，执行完成后可进入报告详情页完成信息绑定与用例入库，并支持历史报告的查看与对比，
**以便**每份报告都有完整的上下文记录，用例沉淀过程清晰可追溯。

**执行完成后的完整流程**：
```
Agent 执行完成
    ↓ 聊天界面推送「查看报告详情」卡片（用户主动点击，不强制跳转）
进入报告详情页
    ├── 在线预览 HTML 报告
    ├── 填写报告名称
    ├── 绑定关联项（自动带入：本次对话、上传文档；可手动补充：业务规则文档、需求文档）
    └── 点击「确认保存并入库」
        ├── 报告保存至报告管理模块
        └── 用例以「草稿」状态写入用例库，自动继承关联信息
```

**报告类型区分**：
- **首次执行报告**：来自系统化 / 探索 / 混沌模式，标注执行模式
- **回归验证报告**：来自 US-005 一键执行，标注"回归验证"及关联的原始用例组

**验收标准**：
- [x] 导航栏新增"测试报告"入口，与"用例库"并列
- [x] 执行完成后，聊天界面出现「查看报告详情」可点击卡片
- [x] 报告 Agent 执行完毕自动保存（无需用户手动点击保存），报告名称自动生成（含模式+日期）
- [x] 报告详情页 4 个 Tab：执行结果（含只看失败开关、入库工具栏）、覆盖分析（coverage_point 树形展开）、关联缺陷、关联信息
- [x] 入库前弹出确认对话框，展示关联文档与首次对话内容，无文档时橙色警告提示
- [x] 入库按钮文字统一为"测试用例入库" / "入库所选 N 条用例"
- [x] 关联信息 Tab：自动列出对话上传文档（磁盘读取，重启安全）、首次用户消息预览、可手动补充额外文档、"查看对话"跳转链接
- [x] 报告自动关联业务文档（uploadedFiles 全链路传递至 report.uploadedDocs）
- [x] 缺陷创建时自动绑定具体执行用例（testCaseId）并嵌入执行日志
- [x] 缺陷详情页分段展示用户描述与执行日志（按 `## 执行日志` 分隔符拆分）
- [x] 报告列表展示：名称、执行模式（含模式筛选）、通过率、用例数、入库状态、保存时间
- [x] 报告列表支持分页、按执行模式过滤、删除操作
- [x] 不再生成思维导图文件
- [ ] 回归验证报告与首次执行报告视觉区分（待完善，当前通过执行模式标签区分）

### US-007：全链路可追溯
**作为**测试工程师或技术负责人，
**我希望**从任意一条缺陷或用例出发，能向上追溯到对应的执行报告、原始对话和业务规则文档，
**以便**在复盘问题、向上汇报或做回归决策时，有完整的上下文支撑，而不是一条孤立记录。

**关联链路**：
```
业务规则文档（上传）
    → 对话（触发 Agent）
        → 测试报告（自动生成，uploadedDocs 自动关联）
            → 执行用例（入库后保留 reportId + conversationId）
                → 缺陷（绑定 testCaseId，嵌入执行日志）
```

**验收标准**：
- [x] 缺陷详情页展示关联用例，用例详情页可跳转来源报告
- [x] 报告"关联信息"Tab 自动展示上传文档和首次对话输入，提供"查看对话"跳转
- [x] 缺陷创建时自动绑定具体执行用例（testCaseId）并记录执行日志，无需手动填写
- [x] 用例入库时自动继承 reportId、conversationId、sourceMode，全链路字段闭合

### US-008：业务规则库（持久化业务文档）
**作为**测试工程师，
**我希望**将业务规范文档（如授信额度测算规则）一次性上传到知识库，并与模型编号绑定，
**以便**后续每次测试对话无需重复上传文档，Agent 自动获取最新业务规则作为上下文。

**验收标准**：
- [x] 知识库管理页"业务规则库"Tab：按 modelId 管理文档，支持新增/编辑/停用/删除
- [x] 对话时输入 modelId，系统自动将该模型的业务规则注入 Agent 上下文
- [x] 支持多版本文档共存，停用旧版本而不删除

### US-009：挡板场景库（命名的外部依赖组合）
**作为**测试工程师，
**我希望**将常用的外部接口挡板配置命名保存（如"黑名单用户"、"卡状态异常"），
**以便**在测试 YAML 中通过名称引用，避免每个用例都手写重复的挡板参数。

**验收标准**：
- [x] 知识库管理页"挡板场景库"Tab：新增/编辑/删除命名场景，setupData 以 JSON 编辑
- [x] YAML `preconditions.external_setup_ref: "场景名"` 可引用命名场景
- [x] 用例内联的 `external_setup` 字段优先级高于场景库（可覆盖部分字段）

### US-010：测试规范库（团队共享测试指令）
**作为**技术负责人，
**我希望**为每个模型/测试模式预设 customInstructions（如"优先覆盖准入拒绝边界"），
**以便**团队成员无需每次在对话中重复描述，保证测试风格统一。

**验收标准**：
- [x] 知识库管理页"测试规范库"Tab：按 modelId+mode 绑定测试规范
- [x] Agent 运行时自动合并规范库中的 customInstructions
- [x] 同一模型可绑定多个模式（systematic/regression/exploratory/chaos），也可设置通用规范（mode=null）

### US-011：覆盖点分类树（标准化词汇表）
**作为**测试工程师，
**我希望**维护一套标准化的覆盖点分类词汇（如"准入条件/月收入"、"系数计算/余额档位"），
**以便**Agent 生成的 `coverage_point` 字段使用统一词汇，报告覆盖分析更清晰。

**验收标准**：
- [x] 知识库管理页"覆盖点分类树"Tab：按 category 分组展示标签，支持批量导入/删除
- [x] Agent 系统提示词自动注入覆盖点词汇表，引导用例生成时使用标准词汇

### US-012：测试数据模板库（字段边界值参考）
**作为**测试工程师，
**我希望**预定义各关键字段的边界值和等价类（如月收入边界：10000、10001、-1），
**以便**Agent 设计用例时直接参考，提高边界覆盖率，减少重复分析。

**验收标准**：
- [x] 知识库管理页"测试数据模板库"Tab：按字段名管理边界值和等价类分区
- [x] Agent 系统提示词自动注入数据模板，辅助用例参数设计

## 七、产品交互流程

### 7.1 AI 测试执行主流程

```
用户在对话框选择测试模式 + 输入测试指令（可上传文档）
              ↓
Agent 按模式 System Prompt 规划测试步骤
              ↓
执行过程中，聊天界面实时展示 test_progress 事件
              ↓
执行完毕，HTML 报告出现在下载面板
              ↓
测试用例自动保存为 DRAFT 入库
```

### 7.2 测试用例晋级流程

```
DRAFT（Agent 自动保存 / 手动创建）
    ↓ 测试工程师 submit
PENDING_REVIEW
    ↓ 审核人 approve
APPROVED
    ↓ 技术负责人 baseline
BASELINE ←─── 可被 AI 回归推荐使用
    ↓ 废弃
DEPRECATED
```

### US-007：缺陷管理
**作为**测试工程师，
**我希望**在测试报告中发现失败用例后，能够一键创建缺陷记录，并通过缺陷管理模块跟踪缺陷的处理进展，
**以便**测试发现的问题有完整的生命周期记录，不遗漏、不重复。

**缺陷生命周期**：
```
待处理 → 处理中 → 已解决 → 已关闭
            ↓
          不修复
```

**与测试报告的关联**：
- 缺陷可关联到测试报告和测试用例，追溯发现来源
- 缺陷详情页展示关联的报告和用例链接

**验收标准**：
- [ ] 导航栏新增"缺陷管理"入口
- [ ] 支持手动创建缺陷（标题、严重级别 P0-P3、描述）
- [ ] 缺陷状态：待处理/处理中/已解决/已关闭/不修复
- [ ] 缺陷详情页支持更新状态、添加评论
- [ ] 缺陷可关联到测试报告和测试用例
- [ ] 缺陷列表支持按状态、严重级别筛选
