## 一、产品概述

### 1.1 背景

传统接口测试流程依赖人工阅读业务规范文档、手工编写测试用例、逐条调用接口验证，效率低且容易遗漏边界场景。

本产品在 TestAgent-PI 平台上新增一套 **AI 驱动的接口测试自动化能力**，用户只需上传业务规范文档，Agent 即可自动解析规则、枚举因子组合、生成测试用例、执行接口测试并输出测试报告。

### 1.2 产品定位

面向测试工程师和后端开发团队，提供"文档驱动的接口测试自动化"能力，将原本需要数天的测试准备工作压缩至分钟级。

### 1.3 核心价值

| 价值点 | 说明 |
|--------|------|
| 全因子覆盖 | 自动枚举业务规则中所有因子组合，不遗漏边界场景 |
| 规则理解准确 | LLM 理解业务语义，正确计算每条用例的预期值 |
| 人工可控 | 生成 JSON 中间层供人工 Review，确认后再执行 |
| 端到端自动化 | 从文档到报告全链路打通，最终复用 HTTPRunner 标准报告 |

## 三、用户故事

### US-001：文档解析生成测试用例（核心）
**作为**测试工程师，
**我希望**上传业务规范文档后，Agent 能自动解析规则并生成完整的因子组合测试用例（JSON 格式），
**以便**我能快速 Review 用例覆盖情况，而不需要手工分析文档。

**验收标准**：
- [ ] 上传业务规范文档后，Agent 识别出所有决策因子
- [ ] 每个因子的所有有效取值都被枚举
- [ ] 每条用例的预期结果（admit_flag、credit_limit）计算正确
- [ ] 输出 `cases_summary.json`，格式符合规范

### US-002：生成可执行测试脚本
**作为**测试工程师，
**我希望**确认 JSON 用例后，Agent 自动为每条用例生成 SQL 预埋文件和 HTTPRunner YAML 测试脚本，
**以便**我可以直接执行测试，无需手工编写脚本。

**验收标准**：
- [ ] 每条用例对应一个 `TC_XXX_setup.sql`（幂等，含 DELETE + INSERT）
- [ ] 每条用例对应一个 `TC_XXX_test.yaml`（HTTPRunner v4 格式）
- [ ] YAML 中的断言与 JSON 预期值一致
- [ ] 文件命名规范，目录结构清晰

### US-003：执行测试并生成报告
**作为**测试工程师，
**我希望**Agent 自动执行 SQL 预埋和 HTTPRunner 测试，并输出测试报告，
**以便**我能直接看到测试结果，无需手动操作。

**验收标准**：
- [ ] Agent 调用 httprunner 执行所有用例
- [ ] 执行完成后输出 HTML 报告（HTTPRunner 原生）
- [ ] 聊天窗口显示 Markdown 摘要（总数、通过、失败、通过率）
- [ ] 失败用例列出失败原因

## 四、功能需求

### 4.1 Step 1：文档解析 → JSON 用例生成

| 需求编号 | 需求描述 | 优先级 |
|---------|---------|-------|
| F-001 | 支持上传 .txt/.md 格式的业务规范文档 | P0 |
| F-002 | Agent 提取准入规则（多条件 AND/OR 逻辑） | P0 |
| F-003 | Agent 提取计算规则（系数、公式） | P0 |
| F-004 | Agent 枚举所有有意义的因子组合（剪枝冗余组合） | P0 |
| F-005 | Agent 为每条组合计算预期结果（admit_flag、credit_limit） | P0 |
| F-006 | 输出 cases_summary.json，格式字段完整 | P0 |
| F-007 | 支持通过模板文件指定用例格式 | P1 |

### 4.2 Step 2：JSON → SQL + YAML 生成

| 需求编号 | 需求描述 | 优先级 |
|---------|---------|-------|
| F-008 | 每条用例生成独立 SQL 文件（DELETE + INSERT，幂等） | P0 |
| F-009 | 每条用例生成独立 HTTPRunner v4 YAML 文件 | P0 |
| F-010 | YAML 断言使用 contains 匹配 XML 响应片段 | P0 |
| F-011 | YAML base_url 使用环境变量 ${ENV(BASE_URL)} | P0 |
| F-012 | 文件按 sql/ 和 testcases/ 两个目录存放 | P1 |

### 4.3 Step 3：执行 + 报告

| 需求编号 | 需求描述 | 优先级 |
|---------|---------|-------|
| F-013 | Agent 通过 bash 工具执行 SQL 预埋 | P0 |
| F-014 | Agent 调用 httprunner run 执行测试 | P0 |
| F-015 | 生成 HTTPRunner HTML 报告 | P0 |
| F-016 | 聊天窗口输出 Markdown 摘要（通过率、失败用例） | P0 |

## 五、非功能需求

| 类型 | 需求 |
|------|------|
| 性能 | 18 条用例全量生成（JSON + SQL + YAML）在 60s 内完成 |
| 准确性 | 预期值计算准确率 100%（基于确定性公式） |
| 幂等性 | SQL 文件可重复执行，不产生脏数据 |
| 可读性 | 生成的文件有注释，人工可读可修改 |
| 兼容性 | HTTPRunner YAML 符合 v4 规范，可直接执行 |

---

## 六、约束与边界

- 业务文档格式：纯文本（.txt / .md），不含 PDF/Word 解析
- 接口协议：XML over HTTP，无鉴权
- 数据库：PostgreSQL，SQL 文件针对 PostgreSQL 语法
- HTTPRunner 版本：v4（Go 版）
- 测试环境：Mock Server（本项目内置）

---

## 七、产品交互流程

```
用户上传文档 + 在对话框输入指令
         ↓
Agent 解析文档（Step 1）
         ↓
输出 cases_summary.json → 用户下载 Review
         ↓
用户输入"已确认，请生成脚本"
         ↓
Agent 生成 SQL + YAML（Step 2）
         ↓
用户输入"请执行测试"（或 Agent 自动执行）
         ↓
执行 SQL 预埋 → HTTPRunner 执行（Step 3）
         ↓
HTML 报告 + 聊天摘要输出
```

## 八、验收标准总览

以《授信额度测算业务规范（MODEL_001）》为验收样本：

| 验收项 | 标准 |
|--------|------|
| 用例数量 | 生成 18 条有效用例（3 不准入 + 15 准入） |
| 预期值准确率 | 18/18 = 100% |
| SQL 可执行率 | 18/18 条 SQL 执行无报错 |
| YAML 可执行率 | 18/18 条 YAML HTTPRunner 执行无格式错误 |
| 接口通过率 | 18/18 = 100%（基于正确的 Mock Server） |
| 报告完整性 | HTML 报告 + Markdown 摘要均正常输出 |
