# 待优化事项

记录已识别但尚未实施的优化点，按优先级排序。

---

## 一、系统提示词通用化（高优先级）

**问题**：`system-prompt.ts` 中存在两类 MODEL001 专属内容，换模型时必须手动修改提示词。

### 1.1 硬编码业务规则摘要

`## MODEL_001 业务规则摘要` 段落把准入条件、系数三档、计算公式写死在提示词里。
应改为：LLM 完全从上传的业务规范文档中读取规则，提示词只保留"你需要从文档中提取这些内容"的指令。

### 1.2 准入条件 Checklist 通用化

当前建议的 Step 2a checklist 写死了 MODEL001 的 6 个字段名：

```
☐ monthly_salary ≤ 10000
☐ social_security_flag = 0
☐ card_status ≠ NORMAL
...
```

应改为通用指令：
> "从业务规范文档中提取所有准入条件，列成清单，确保每条条件至少有一个独立的 FAIL 用例。"

### 1.3 系数档显式声明通用化

当前建议的系数档声明绑定了 MODEL001 的三档规则（`avg_3m_balance > 1000 → 2.3` 等）。
应改为通用指令：
> "从业务规范文档中找到系数/乘数的分档规则，写出本用例适用哪一档及判断依据，再代入 expression。"

## 二、Agent 用例质量加固（中优先级）

**问题**：Agent 可能生成"看起来正确但实际有缺陷"的测试用例，导致漏 Bug。

### 2.1 准入判断显式推理（风险一：LLM 与 Bug 犯同一个错）

**场景**：8001 的 Bug 是 `salary >= 10000` 误放行。若 LLM 也误认为 10000 应 PASS，测试绿了但 Bug 被漏掉。

**修改方向**：Step 2a 要求对每个用例显式写出准入判断过程，例如：
```
TC_S05 | salary=10000
  → monthly_salary > 10000? 10000 > 10000 = false → 拒绝
  → expected_admission: false
```

### 2.2 系数档选择显式声明（风险二：系数档选错）

**场景**：avg=1000 应取系数 0.5，LLM 用了 2.3，算出的 expectedResult 偏大 4.6 倍。

**修改方向**：Step 2b 调 `calculate_value` 前，要求先写：
```
avg=1000 → 系数档：0 < 1000 ≤ 1000 → coefficient=0.5
expression: "2 * (1000/1000) * 12000 * 0.5"
```

### 2.3 preconditions 与 expression 数值一致性（风险三）

**场景**：YAML 里 `avg_3m_balance: 1500`，但 expression 用了 `1000`，用例本身写错。

**修改方向**：Step 2c 加规则：expression 中每个数值必须能在同一用例的 `preconditions` 中找到对应字段。

### 2.4 准入条件独立覆盖 Checklist（风险四）

**场景**：LLM 只覆盖"全部满足→通过"和"全部不满足→拒绝"，漏掉"只有第 N 条不满足→拒绝"。

**修改方向**：Step 2a 末尾加强制 checklist（通用版，见 1.2），未全部勾选不能进 Step 2b。

## 三、server 侧预校验（低优先级，可选）

**问题**：即使提示词加固后，仍可能有 LLM 跳过步骤的情况，缺乏兜底。

**修改方向**：`run_test_suite` 执行前，对有 `credit_limit` assertion 的用例，从 `preconditions` 取值重算期望值，若偏差 > 0.01 则返回"用例设计错误"而非执行测试。

**代价**：需在 `test-runner.ts` 中内置 MODEL001 公式，与被测系统耦合。待一、二项完成后再评估是否值得做。
