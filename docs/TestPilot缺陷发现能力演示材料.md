## 一、演示背景

### 被测接口

| 项目 | 说明 |
|------|------|
| 接口名称 | MODEL_001 授信额度测算 |
| 接口地址 | POST /mock/model/score |
| 核心逻辑 | 根据社保、工资、资产、外部指标判断是否准入并计算授信额度 |

### 准入规则（6 条全满足才准入）

| # | 条件 | 类型 |
|---|------|------|
| 1 | monthly_salary > 10000（严格大于） | 本地指标 |
| 2 | social_security_flag = 1 | 本地指标 |
| 3 | card_status = "NORMAL" | 外部指标 |
| 4 | id_check_result = "PASS" | 外部指标 |
| 5 | is_black = false | 外部指标 |
| 6 | recent_trans_amount > 0 | 外部指标 |

### 额度公式

credit_limit = user_level × **(avg_3m_balance ÷ 1000)** × monthly_salary × coefficient

| avg_3m_balance 范围 | coefficient |
|---------------------|-------------|
| > 1000              | 2.3         |
| 0 < balance ≤ 1000  | 0.5         |
| ≤ 0（含负数）       | 0.2         |

### 发现的缺陷详情

#### Bug-1：外部指标准入检查缺失（高危）

- **现象**：card_status 异常、命中黑名单、身份证核验失败、无近期交易时，系统仍错误判断为准入
- **业务规则**：6 条准入条件全部满足才准入，其中 4 条为外部指标
- **系统实现**：外部指标检查被完全删除，仅保留月薪和社保 2 条
- **影响**：高风险客户（黑名单、异常卡状态）被错误授信，信贷风险极高

| 用例名称 | 失败断言 |
|----------|---------|
| 不准入-借记卡状态异常 | admit_flag: expected=0 actual=1，credit_limit: expected=0.00 actual=690000000.00 |
| 不准入-命中黑名单 | admit_flag: expected=0 actual=1，credit_limit: expected=0.00 actual=1656000000.00 |
| 不准入-身份证核验失败 | admit_flag: expected=0 actual=1，credit_limit: expected=0.00 actual=207000000.00 |
| 不准入-无近期交易 | admit_flag: expected=0 actual=1，credit_limit: expected=0.00 actual=1380000000.00 |

---

#### Bug-2：准入边界判断错误（高危）

- **现象**：月薪恰好等于 10000 时，系统错误判断为准入
- **业务规则**：monthly_salary > 10000（严格大于）
- **系统实现**：monthly_salary >= 10000（大于等于）
- **影响**：月薪恰好 10000 的边界客户被错误授信

| 用例名称 | 失败断言 |
|----------|---------|
| 不准入-工资不足10000（值=10000） | admit_flag: expected=0 actual=1 |
| 边界值-工资刚好不准入10000 | admit_flag: expected=0 actual=1，credit_limit: expected=0.00 actual=69000000.00 |

---

#### Bug-3：授信额度公式双重错误（高危）

- **现象 1**：avg_3m_balance 未除以 1000，直接以元值参与计算，结果偏大约 1000 倍
- **现象 2**：系数档位从 3 档简化为 2 档（0 < balance ≤ 1000 时应用 0.5，实际用了 2.3）
- **业务规则**：三档系数：>1000→2.3；0<x≤1000→0.5；≤0→0.2，且 balance 需 ÷1000
- **系统实现**：只有两档：>0→2.3；≤0→0.5，且 balance 未 ÷1000
- **影响**：所有正资产准入客户的授信额度严重偏大，资金风险极高

| 用例名称 | 期望额度 | 实际额度 | 偏差说明 |
|----------|----------|----------|---------|
| 正常准入-高资产-系数2.3 | 103,500.00 | 103,500,000.00 | ÷1000 缺失，偏大 1000 倍 |
| 正常准入-中等资产-系数0.5 | 7,500.00 | 34,500,000.00 | ÷1000 缺失 + 系数用 2.3 而非 0.5，偏大 4600 倍 |
| 边界值-高资产边界1001-系数2.3 | 103,603.50 | 103,603,500.00 | ÷1000 缺失，偏大 1000 倍 |
| 边界值-中等资产边界1000-系数0.5 | 22,500.00 | 103,500,000.00 | ÷1000 缺失 + 系数用 2.3 而非 0.5 |
| 特殊值-monthly_salary极大值 | 23,000,000.00 | 23,000,000,000.00 | ÷1000 缺失，偏大 1000 倍 |

## 四、方案 Y：现场演示操作手册

> 演示前确认：双击启动台中的 TestPilot，等待服务就绪（终端显示 5/5 全部 OK）

### 步骤 1 — 打开 TestPilot

访问 http://localhost:3000，登录后进入聊天界面。

---

### 步骤 2 — 新建对话，上传两个文件

点击「新建对话」，上传：
- MODEL001_业务规则文档(LLM 友好格式).md
- MODEL001_用例样例.yaml

---

### 步骤 3 — 发送以下消息（直接复制粘贴）

```
请分析上传的业务规则文档，完成以下测试工作（生成用例时参照同时上传的 MODEL001_用例样例.yaml 格式示例）：

1. 仔细阅读业务规则文档，提取所有输入参数、业务规则、计算公式
2. 基于等价类划分和边界值分析，生成完整的接口测试用例（YAML 格式，严格参照 MODEL001_用例样例.yaml），每条用例必须包含 coverage_point 字段，描述该用例的覆盖点
3. 执行所有测试用例。注意：被测系统地址是 http://localhost:8001

执行完成后报告将自动保存，点击对话中的「查看报告详情」卡片可查看完整执行结果，并在报告详情页进行用例入库、缺陷登记等操作
```

---

### 步骤 4 — 等待 Agent 执行（约 2~3 分钟）

1. 读取业务规则文档，提取规则和公式

3. 并发执行所有用例：写入 DB 前置数据 → 调用接口 → 验证断言
