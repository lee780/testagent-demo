## 一、测试用例生命周期状态机

### 1.1 状态定义

| 状态 | 含义 | 可操作人 |
|------|------|---------|
| `DRAFT` | 草稿，可修改 | 测试工程师 |
| `PENDING_REVIEW` | 已提交，等待审核 | （只读，等待审核人） |
| `APPROVED` | 审核通过 | 审核人审批 |
| `BASELINE` | 已基线，作为回归推荐候选 | 技术负责人晋级 |
| `DEPRECATED` | 已废弃，不再使用 | 任何有权限的用户 |

### 1.2 状态流转图

```
              submit()
DRAFT ──────────────────────▶ PENDING_REVIEW
  ▲                                │
  │         reject(reviewNote)     │ approve(userId, reviewNote)
  └────────────────────────────────┘
                                   │
                                   ▼
                              APPROVED
                                   │
                          baseline()│
                                   ▼
                              BASELINE ────▶ DEPRECATED
                                   │          ▲
                (修改已审批/基线用例时) │          │
                创建新版本 (version+1)  │         deprecate()
                新版本为 DRAFT          │          │
                                        └──────────┘
```

### 1.3 版本控制设计

- `TestCaseGroup`：记录用例的逻辑身份，`(modelId, caseCode)` 唯一约束
- `TestCase`：每次创建或修改产生新快照，`(groupId, version)` 唯一约束
- `TestCaseGroup.latestId`：始终指向该组最新版本的 TestCase.id
- 修改 APPROVED 或 BASELINE 用例时：
  1. 旧 TestCase 状态不变（保留历史）
  2. 新建 TestCase，version = 旧 version + 1，status = DRAFT
  3. 更新 TestCaseGroup.latestId 指向新 TestCase

### 1.4 关键 service 方法（testcase.service.ts）

```typescript
// 提交审核：DRAFT → PENDING_REVIEW
async function submitForReview(id: string): Promise<TestCase>

// 审批：PENDING_REVIEW → APPROVED（或拒绝回 DRAFT）
async function approveTestCase(id: string, reviewerId: string, reviewNote?: string): Promise<TestCase>
async function rejectTestCase(id: string, reviewerId: string, reviewNote: string): Promise<TestCase>

// 晋级基线：APPROVED → BASELINE
async function baselineTestCase(id: string): Promise<TestCase>

// AI 回归推荐：返回按相关度排序的 BASELINE 用例
async function recommendTestCases(params: { description: string, modelId?: string, limit?: number }): Promise<RecommendResult[]>

// 版本历史
async function getVersionHistory(groupId: string): Promise<TestCase[]>

// 执行历史
async function getExecutionHistory(testCaseId: string, limit: number): Promise<TestExecution[]>
```

## 二、Agent System Prompt 设计

### 3.1 四种模式 Prompt 约束（system-prompt.ts）

| 模式 | 关键约束 | 必须包含关键词 |
|------|---------|--------------|
| `regression` | 禁止 LLM 生成用例，只运行现有 YAML | "REGRESSION MODE"、"FORBIDDEN"、"NEVER generate" |
| `systematic` | BVA 驱动，用例 ID 顺序编号，禁止随机值 | "SYSTEMATIC MODE"、"BVA"、"NEVER use random values" |
| `exploratory` | 假设驱动，每次执行后自适应下一步 | "EXPLORATORY MODE"、"Hypothesis-Driven" |
| `chaos` | 同一用例集跑两套系统，差异分析 | "CHAOS"、"localhost:8000"、"localhost:8001" |

## 四、测试报告与用例手动入库设计

### 4.1 执行后数据流（saveRawTestCases）

Agent 执行测试完毕后，**不再自动写入 TestCase 表**，而是将原始用例数据写入文件：

```
src/agent-core/tools/test-runner.ts
    ↓ 测试执行完毕
    ↓ saveRawTestCases(allJobs.map(j => j.tc), config.outputDir, config.userId, config.conversationId)
    ↓ mkdirSync({outputDir}/{userId}/{conversationId})
    ↓ writeFileSync('test_cases_raw.json', JSON.stringify(cases))
```

### 4.2 报告保存流程（report.service.ts）

```
POST /api/reports  ← 用户在下载面板点击"保存到报告库"
    ↓ createReport()
    ↓ 读取 test_cases_raw.json → 存入 TestReport.testCasesData（JSON）
    ↓ INSERT test_reports（casesImported=false）
```

### 4.3 用例入库流程（手动触发）

```
POST /api/reports/:id/import  ← 用户在报告列表点击"确认入库"
    ↓ importTestCases()
    ↓ 读取 TestReport.testCasesData（JSON 数组）
    ↓ for each case:
    │   1. 从 tc.id 解析 modelId（TC_MODEL001_S01 → MODEL001）
    │   2. 查找或创建 TestCaseGroup（modelId + caseCode）
    │   3. 计算版本号（lastVersion + 1）
    │   4. 创建 TestCase（status=DRAFT, version, sourceMode, conversationId, reportId）
    │   5. 更新 TestCaseGroup.latestId
    ↓ UPDATE test_reports SET casesImported = true
```

### 4.4 用例数据映射

test_cases_raw.json 字段 → TestCase 字段：

| JSON 字段 | TestCase 字段 | 说明 |
|----------|--------------|------|
| id（如 TC_MODEL001_S01） | group.modelId + group.caseCode | 解析模型ID和用例编码 |
| name | title | 用例名称 |
| coverage_point | coveragePoint | 覆盖点描述 |
| priority | priority | 优先级 P0-P3 |
| request | inputParams（JSON） | 输入参数 |
| assertions | expectedResult（JSON） | 预期结果 |
| report.executionMode | sourceMode | 来源模式（继承自报告） |
| report.conversationId | conversationId | 来源对话 |
| report.id | reportId | 关联报告 |

## 五、AI 回归推荐算法

### 5.1 推荐逻辑（testcase.service.ts recommendTestCases）

基于大模型语义推理，无需 Embedding 服务：

```
1. 查询所有 BASELINE 状态最新版本用例（按 modelId 过滤，如提供）
2. 关联查询每条用例的 TestReport（report.name, executionMode, uploadedDocs）和 Conversation
3. 为每条用例构建上下文字符串：
   [i] modelId/caseCode: "title"
     覆盖点: ...
     来源模式: ...
     关联报告: report.name（executionMode）
     关联文档: doc1, doc2, ...
     关联对话: conversation.title
4. 将变更描述 + 全部用例上下文拼入 Prompt，调用 LLM 返回 JSON 数组：
   [{"index": 0, "score": 95, "reason": "直接覆盖变更涉及的计算逻辑..."}]
5. 按 score 降序排序，返回前 limit 条（默认 20）
6. LLM 不可达时，降级返回全部基线用例（score=50, reason="LLM 服务暂不可用"）
```

```typescript
interface RecommendResult {
  testCase: TestCase
  score: number   // 0-100 相关度分数（LLM 评分）
  reason: string  // LLM 给出的推荐理由
}
```

## 六、Mock Service 详细设计

### 6.1 calculateCreditScore() 业务逻辑

```
输入：userId
      │
      ▼
并发查询 4 张表（Promise.all）
      │
      ├── 任一表无记录 → result_code=0001（用户不存在）
      │
      ▼
准入判断：
  monthly_salary > 10000 AND social_security_flag = 1
      │
      ├── 不满足 → admit_flag=0, credit_limit=0.00
      │
      ▼ 满足（准入）
系数取值：
  avg_3m_balance > 0 → coefficient = 2.3
  avg_3m_balance ≤ 0 → coefficient = 0.5
      │
      ▼
额度计算：
  crgln = user_level × (avg_3m_balance / 1000) × monthly_salary × coefficient
  crgln < 0 → crgln = 0
      │
      ▼
返回 admit_flag=1, credit_limit=crgln.toFixed(2), result_code=0000
```

关键注意点：
- `avg_3m_balance` 存储单位为元，公式中除以 1000 转换为千元单位
- `toFixed(2)` 保证两位小数格式
- 负额度归零（Math.max(0, crgln)）

## 七、认证中间件设计

### 7.1 authenticate() 逻辑（auth.ts）

```
请求携带 Authorization: Bearer <token>
      │
      ▼
验证 JWT 签名 + 过期时间
      │
      ├── 无效 → 401
      │
      ▼
从 JWT payload 提取 jti（JWT ID）
      │
      ▼
从 Redis 查询 session:{jti}
      │
      ├── sessionData 不为 null → 校验 jti 匹配 → 放行
      ├── sessionData 为 null → [已知 Bug] 当前放行，应改为拒绝
      │
      ▼
request.currentUser = { user_id, username, role }
```

**已知 Bug（待修复）**：`if (sessionData)` 条件导致 Redis key 删除后（注销/密码修改后）Token 仍然有效。修复方式：session 不存在时应返回 401，而非放行。

```
.testagent/workspace/
├── sessions/         # pi-mono Agent 会话文件（按 conversationId）
├── scratch/          # Agent 临时工作区
└── outputs/
    └── {userId}/
        └── {conversationId}/
```

### 8.2 下载接口实现

- 路径：`GET /api/conversations/:id/outputs/:filename`
- 验证：conversationId 归属当前用户
- 响应：`Content-Type: application/octet-stream`，`Content-Disposition: attachment; filename=...`
- 前端预览 HTML：`new Blob([text], { type: 'text/html' })` → `URL.createObjectURL` → `window.open`
