## 一、测试用例生命周期状态机

### 1.1 状态定义

| 状态 | 含义 | 可操作人 |
|------|------|---------|
| `DRAFT` | 草稿，可修改 | 测试工程师 |
| `PENDING_REVIEW` | 已提交，等待审核 | （只读，等待审核人） |
| `APPROVED` | 审核通过 | 审核人审批 |
| `BASELINE` | 已基线，作为回归推荐候选 | 技术负责人晋级 |
| `DEPRECATED` | 已废弃，不再使用 | 任何有权限的用户 |

### 1.2 状态流转图

```
              submit()
DRAFT ──────────────────────▶ PENDING_REVIEW
  ▲                                │
  │         reject(reviewNote)     │ approve(userId, reviewNote)
  └────────────────────────────────┘
                                   │
                                   ▼
                              APPROVED
                                   │
                          baseline()│
                                   ▼
                              BASELINE ────▶ DEPRECATED
                                   │          ▲
                (修改已审批/基线用例时) │          │
                创建新版本 (version+1)  │         deprecate()
                新版本为 DRAFT          │          │
                                        └──────────┘
```

### 1.3 版本控制设计

- `TestCaseGroup`：记录用例的逻辑身份，`(modelId, caseCode)` 唯一约束
- `TestCase`：每次创建或修改产生新快照，`(groupId, version)` 唯一约束
- `TestCaseGroup.latestId`：始终指向该组最新版本的 TestCase.id
- 修改 APPROVED 或 BASELINE 用例时：
  1. 旧 TestCase 状态不变（保留历史）
  2. 新建 TestCase，version = 旧 version + 1，status = DRAFT
  3. 更新 TestCaseGroup.latestId 指向新 TestCase

### 1.4 关键 service 方法（testcase.service.ts）

```typescript
// 提交审核：DRAFT → PENDING_REVIEW
async function submitForReview(id: string): Promise<TestCase>

// 审批：PENDING_REVIEW → APPROVED（或拒绝回 DRAFT）
async function approveTestCase(id: string, reviewerId: string, reviewNote?: string): Promise<TestCase>
async function rejectTestCase(id: string, reviewerId: string, reviewNote: string): Promise<TestCase>

// 晋级基线：APPROVED → BASELINE
async function baselineTestCase(id: string): Promise<TestCase>

// AI 回归推荐：返回按相关度排序的 BASELINE 用例
async function recommendTestCases(params: { description: string, modelId?: string, limit?: number }): Promise<RecommendResult[]>

// 版本历史
async function getVersionHistory(groupId: string): Promise<TestCase[]>

// 执行历史
async function getExecutionHistory(testCaseId: string, limit: number): Promise<TestExecution[]>
```

## 四、测试报告与用例手动入库设计

### 4.1 执行后数据流（run_test_suite 自动建报告）

Agent 执行测试完毕后，**直接在 DB 创建 TestReport 记录**（无需用户手动保存）：

```
src/agent-core/tools/test-runner.ts
    ↓ 测试执行完毕（runWithConcurrency 结束）
    ↓ 计算 passed/failed/errors/recommendation
    ↓ 组装报告名称：`${suiteName} [${modeLabel}] ${MM-DD}`
    ↓ prisma.testReport.create({
         name: reportName,
         executionMode: config.mode ?? "systematic",
         executionResults: allResults,    // 每条用例执行明细（JSON）
         testCasesData: allJobs.map(j => j.tc),  // 原始 YAML 用例（JSON）
         recommendation,
         uploadedDocs: config.uploadedFiles ?? [],  // 自动从 chat.service 传入
         casesImported: false             // 等待用户手动确认入库
       })
    ↓ 返回 report_id 和 report_url 给 Agent（渲染为可点击卡片）
```

报告名称格式示例：`信用评分接口测试 [系统化] 03-22`

**uploadedFiles 传递链路**（v2.4 新增）：
```
POST /api/chat/stream
    → chatFilesIndex.get(conversationId)  // 当次会话内存索引
    → sendMessageStreaming({ uploadedFiles })
    → runTestAgent({ uploadedFiles })
    → createToolConfig({ uploadedFiles })  // 写入 ToolConfig
    → test-runner auto-create report: uploadedDocs = config.uploadedFiles
```

用户上传的文件同时**持久化到磁盘**：`./storage/chat/{userId}/{conversationId}/`，通过 `GET /api/conversations/:id/uploads` 读取，服务重启后仍准确。

### 4.2 报告管理 API（report.service.ts）

```
GET  /api/reports?page=&pageSize=&mode=   ← 列表，支持执行模式过滤
GET  /api/reports/:id                     ← 详情（含 executionResults、testCasesData、testCases）
PUT  /api/reports/:id                     ← 更新名称/关联文档
DELETE /api/reports/:id                   ← 删除报告
     ↓ deleteReport()
     ↓ updateMany TestCase SET reportId=null WHERE reportId=:id
     ↓ updateMany Defect  SET reportId=null WHERE reportId=:id
     ↓ testReport.delete({ id })
     （先清空 FK 引用，再删除，无需级联配置）
```

### 4.3 用例入库流程（手动触发，支持部分选择）

```
POST /api/reports/:id/import  ← body: { caseIds?: string[] }（省略则全部入库）
    ↓ importTestCases(reportId, userId, caseIds?)
    ↓ 读取 TestReport.testCasesData（JSON 数组）
    ↓ 按 caseIds 过滤（若提供）
    ↓ for each case（每条独立事务）：
    │   prisma.$transaction(async tx => {
    │     1. 从 tc.id 解析 modelId（TC_MODEL001_S01 → MODEL001）
    │     2. tx.testCaseGroup.findUnique 或 create（modelId + caseCode）
    │     3. 查 lastVersion → version = lastVersion + 1
    │     4. tx.testCase.create（status=DRAFT, version, notes, tags, sourceMode,
    │                           conversationId, reportId, yamlContent）
    │     5. tx.testCaseGroup.update({ latestId: created.id })
    │   })  — 每条原子，单条失败不影响其他条
    ↓ UPDATE test_reports SET casesImported = true
    ↓ 返回 { imported: N, ids: [...], failed: [...] }
```

## 五、AI 回归推荐算法

### 5.1 推荐逻辑（testcase.service.ts recommendTestCases）

基于大模型语义推理，无需 Embedding 服务：

```
1. 查询所有 BASELINE 状态最新版本用例（按 modelId 过滤，如提供）
2. 关联查询每条用例的 TestReport（report.name, executionMode, uploadedDocs）和 Conversation
3. 为每条用例构建上下文字符串：
   [i] modelId/caseCode: "title"
     覆盖点: ...
     来源模式: ...
     关联报告: report.name（executionMode）
     关联文档: doc1, doc2, ...
     关联对话: conversation.title
4. 将变更描述 + 全部用例上下文拼入 Prompt，调用 LLM 返回 JSON 数组：
   [{"index": 0, "score": 95, "reason": "直接覆盖变更涉及的计算逻辑..."}]
5. 按 score 降序排序，返回前 limit 条（默认 20）
6. LLM 不可达时，降级返回全部基线用例（score=50, reason="LLM 服务暂不可用"）
```

```typescript
interface RecommendResult {
  testCase: TestCase
  score: number   // 0-100 相关度分数（LLM 评分）
  reason: string  // LLM 给出的推荐理由
}
```

## 六、Mock Service 详细设计

### 6.1 calculateCreditScore() 业务逻辑（v2.0）

```
输入：userId
      │
      ▼
并发查询 4 张本地DB表（Promise.all）
+ 查询 MockExternalIndicator 表（外部接口挡板）
      │
      ├── 任一表无记录 → result_code=0001（用户不存在）
      │
      ▼
准入判断（6条，全部满足才准入）：
  1. monthly_salary > 10000
  2. social_security_flag = 1
  3. card_status = "NORMAL"
  4. id_check_result = "PASS"
  5. is_black = false
  6. recent_trans_amount > 0
      │
      ├── 任一不满足 → admit_flag=0, credit_limit=0.00
      │
      ▼ 全部满足（准入）
系数取值（3档）：
  avg_3m_balance > 1000       → coefficient = 2.3
  0 < avg_3m_balance ≤ 1000   → coefficient = 0.5
  avg_3m_balance ≤ 0          → coefficient = 0.2
      │
      ▼
额度计算：
  crgln = user_level × avg_3m_balance × monthly_salary × coefficient
  crgln < 0 → crgln = 0
      │
      ▼
返回 admit_flag=1, credit_limit=crgln.toFixed(2), result_code=0000
```

关键注意点：
- `avg_3m_balance` 直接参与计算（v2.0 去掉 ÷1000）
- 外部指标通过 `MockExternalIndicator` 挡板表读取，不调真实第三方
- `toFixed(2)` 保证两位小数格式
- 负额度归零（Math.max(0, crgln)）

### 6.2 外部接口挡板设计

测试前通过 `preconditions.external_setup` 写入 `MockExternalIndicator` 表，mock service 直接读取，不发起真实外部调用。

| 字段 | 来源接口 | 说明 |
|------|---------|------|
| `card_status` | A00110001 借记卡查询 | `NORMAL` / `ABNORMAL` |
| `recent_trans_amount` | A00110001 借记卡查询 | 近30天交易金额，>0 表示有交易 |
| `id_check_result` | A00110002 身份核验 | `PASS` / `FAIL` |
| `is_black` | A00110003 黑名单校验 | `true` / `false` |

`run_test_suite` 执行时，若 YAML 中含 `preconditions.external_setup`，自动 upsert 到 `MockExternalIndicator` 表（user_id 取自 `db_setup.user_info.user_id`）。

## 七、认证中间件设计

### 7.1 authenticate() 逻辑（auth.ts）

```
请求携带 Authorization: Bearer <token>
      │
      ▼
验证 JWT 签名 + 过期时间
      │
      ├── 无效 → 401
      │
      ▼
从 JWT payload 提取 jti（JWT ID）
      │
      ▼
从 Redis 查询 session:{jti}
      │
      ├── sessionData 不为 null → 校验 jti 匹配 → 放行
      ├── sessionData 为 null → 401（已修复）
      │
      ▼
request.currentUser = { user_id, username, role }
```

**已修复（v2.0）**：原 `if (sessionData)` 条件导致 Redis key 删除后 Token 仍然有效（BUG-001），现已修正为 `sessionData 为 null → 401 会话已失效`，注销和密码修改后旧 Token 立即失效。

## 八、可靠性与安全设计（v2.3 新增）

### 8.1 SSE 连接保活与超时配置

| 参数 | 环境变量 | 默认值 | 说明 |
|------|---------|-------|------|
| 心跳间隔 | `SSE_HEARTBEAT_INTERVAL_MS` | 15000 (15s) | 定时向队列推送 heartbeat 保持连接 |
| 空闲超时 | `SSE_IDLE_TIMEOUT_MS` | 120000 (120s) | 超过此时间无事件则关闭连接 |

**启动检查**：若 `HEARTBEAT_INTERVAL_MS ≥ IDLE_TIMEOUT_MS`，服务启动时抛出异常（fail-fast），防止配置错误导致运行期问题。

### 8.2 文件下载路径安全

下载接口 `GET /api/conversations/:id/outputs/:filename` 使用 `path.relative()` 验证路径穿越：

```typescript
const rel = path.relative(baseDir, filePath);
if (rel.startsWith('..') || path.isAbsolute(rel)) {
  return reply.code(403).send({ error: 'Forbidden' });
}
```

替代了旧版不安全的 `filePath.startsWith(baseDir + path.sep)` 判断。

### 8.3 数据库性能优化

- `Conversation` 表：`@@index([userId, updatedAt])` 复合索引，支持 `WHERE userId ORDER BY updatedAt DESC` 高效查询（原 `@@index([userId])` 单列索引无法利用排序）
- 消息历史加载上限：200 条（原 1000 条），防止大型对话加载 OOM

### 12.4 Agent 上下文注入流程

```
前端发送 POST /api/chat/stream { modelId: "MODEL_001", mode: "systematic", ... }
    ↓
chat.service.ts → loadKnowledgeContext(modelId, mode)
    ├── 查询 knowledge_docs (isActive=true) → businessRulesMd
    ├── 查询 test_specs (mode 优先精确匹配，fallback 到 mode=null) → customInstructions
    ├── 查询 coverage_tags → coverageTags[]
    └── 查询 test_data_templates → dataTemplates[]
    ↓
构建用户消息时注入 [SYSTEM CONTEXT] 块：
    ├── ## 业务规则（businessRulesMd）
    ├── ## 覆盖点词汇表（coverageTags 格式化为分类树）
    └── ## 测试数据参考（dataTemplates 格式化为字段边界表）
    ↓
runTestAgent() 接收 customInstructions → buildSystemPrompt()
    └── 注入到 ## 【团队测试规范】 段落
```

### 12.5 挡板场景引用机制

YAML 测试用例的 `preconditions` 支持两种方式配置外部挡板：

```yaml
# 方式一：直接内联（inline）
preconditions:
  external_setup:
    card_status: ABNORMAL
    is_black: true

# 方式二：引用命名场景
preconditions:
  external_setup_ref: "黑名单用户"       # 从 mock_scenes 表按 name 查找
  external_setup:                        # 可选：覆盖场景中的部分字段
    recent_trans_amount: 5000

# 合并优先级：external_setup（内联）> external_setup_ref（场景库）
```

`test-runner.ts` 在执行每条用例前解析 `external_setup_ref`，若场景不存在则抛出错误，终止本条用例执行。
