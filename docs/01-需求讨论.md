## 一、需求背景

TestAgent-PI 需要支持以下完整流程：

```
业务规则文档 → 生成测试用例(人工Review) → 执行接口测试 → 输出测试报告
```

以《授信额度测算业务规范》为参考样本，整体分四步：

```
Step 1：上传业务文档 → Agent 解析规则 → 输出 JSON 中间层（供人工 Review）
Step 2：人工确认 JSON → Agent 生成 SQL 预埋文件 + HTTPRunner YAML（各一个）
Step 3：执行 SQL 预埋 → 执行 HTTPRunner
Step 4：复用 HTTPRunner HTML 报告
```

## 三、业务样本分析（MODEL_001 授信额度测算）

### 数据表与关键字段

| 表名 | 字段 | 类型 | 说明 |
|------|------|------|------|
| user_info | user_level | int(1~5) | 用户等级 |
| account_balance | avg_3m_balance | decimal | 近3月月均资产，可为负 |
| cgs_social_security | social_security_flag | int(0/1) | 0=无社保, 1=有社保 |
| salary_summary | monthly_salary | decimal | 近1月工资，范围 1~10000000 |

### 业务规则

**准入判断**（必须同时满足）：
- `monthly_salary > 10000`
- `social_security_flag = 1`

**系数取值**：
- 准入 且 `avg_3m_balance > 0` → 系数 = 2.3
- 准入 且 `avg_3m_balance ≤ 0` → 系数 = 0.5
- 不准入 → 系数 = 0，额度 = 0

**额度公式**：
```
crgln = user_level × avg_3m_balance × monthly_salary × coefficient
crgln < 0 时归零返回 0.00
```

### 因子组合与有效用例数

| monthly_salary | social_security_flag | 是否准入 | avg_3m_balance | user_level | 用例数 |
|---------------|---------------------|---------|---------------|-----------|-------|
| ≤10000 | 0 | 否 | 任意 | 任意 | 1条（代表值） |
| ≤10000 | 1 | 否 | 任意 | 任意 | 1条（代表值） |
| >10000 | 0 | 否 | 任意 | 任意 | 1条（代表值） |
| >10000 | 1 | 是 | <0 | 1~5 | 5条 |
| >10000 | 1 | 是 | =0 | 1~5 | 5条 |
| >10000 | 1 | 是 | >0 | 1~5 | 5条 |

**有效用例合计：18 条**

## 四、完整流程详细设计

### Step 1：Agent 解析文档 → 输出 JSON 中间层

**用户输入**（聊天框）：
> "这是一份技术文档《授信额度测算业务规范》，请根据技术文档生成完整的接口案例，所有的因子组合都进行输出，每条案例一个文件，案例格式按照 test_case_template.yaml 输出"
> 同时上传：业务规范文档 + test_case_template.yaml

**Agent 执行**：
1. 读取业务文档，提取准入规则、系数规则、额度公式、接口报文格式
2. 穷举所有有意义的因子组合，对每条计算预期结果
3. 生成 JSON 中间文件（供人工 Review）：`cases_summary.json`

**JSON 中间层格式**（参照 cases_batch_1.json）：
```json
[
  {
    "id": "TC_MODEL001_001",
    "name": "工资≤10000且无社保不准入",
    "category": "正常流程",
    "preconditions": {
      "user_level": 1,
      "avg_3m_balance": 0,
      "social_security_flag": 0,
      "monthly_salary": 8000
    },
    "expected": {
      "admit_flag": "0",
      "credit_limit": "0.00",
      "coefficient": 0
    },
    "business_rules": ["工资≤10000", "无社保"]
  }
]
```

**输出**：`cases_summary.json`（通过 write_output_file 写出，用户可下载 Review）

---

### Step 2：人工 Review → Agent 生成执行脚本

**人工 Review 要点**：
- 用例覆盖是否完整（有没有漏掉的因子组合）
- 预期结果计算是否正确（admit_flag、credit_limit 数值）
- 用例命名是否清晰

**确认后，用户输入**：
> "JSON 已确认，请生成所有用例的 SQL 预埋文件和 HTTPRunner 测试脚本"

**Agent 为每条用例生成两个文件**：

#### 文件 1：`TC_MODEL001_001_setup.sql`

```sql
-- 用例：TC_MODEL001_001 工资≤10000且无社保不准入
-- 执行时间：测试前

-- 清理旧数据
DELETE FROM user_info WHERE user_id = 'TC001';
DELETE FROM account_balance WHERE user_id = 'TC001';
DELETE FROM cgs_social_security WHERE user_id = 'TC001';
DELETE FROM salary_summary WHERE user_id = 'TC001';

-- 写入测试数据
INSERT INTO user_info (user_id, user_level) VALUES ('TC001', 1);
INSERT INTO account_balance (user_id, avg_3m_balance) VALUES ('TC001', 0);
INSERT INTO cgs_social_security (user_id, social_security_flag) VALUES ('TC001', 0);
INSERT INTO salary_summary (user_id, monthly_salary) VALUES ('TC001', 8000);
```

#### 文件 2：`TC_MODEL001_001_test.yaml`（HTTPRunner 原生格式）

```yaml
config:
  name: "TC_MODEL001_001 工资≤10000且无社保不准入"
  base_url: "${ENV(BASE_URL)}"

teststeps:
  - name: "调用授信测算接口"
    request:
      url: /api/model/score
      method: POST
      headers:
        Content-Type: application/xml;charset=utf-8
      body: |
        <?xml version="1.0" encoding="utf-8"?>
        <REQUEST>
          <user_id>TC001</user_id>
          <model_id>MODEL_001</model_id>
          <channel>131</channel>
          <business_params>
            <sx_trace_id>TC001_TRACE_001</sx_trace_id>
            <corp_id></corp_id>
            <corp_xy_num></corp_xy_num>
            <person_idtype>1010</person_idtype>
            <person_id></person_id>
            <person_name></person_name>
            <bnk_no></bnk_no>
            <cp_code>8890</cp_code>
            <c_code>110110</c_code>
            <p_code>110000</p_code>
            <fr_psn_id></fr_psn_id>
            <fr_psn_name></fr_psn_name>
          </business_params>
        </REQUEST>
    validate:
      - eq: ["status_code", 200]
      - contains: ["body", "<result_code>0000</result_code>"]
      - contains: ["body", "<admit_flag>0</admit_flag>"]
      - contains: ["body", "<credit_limit>0.00</credit_limit>"]
```

> ⚠️ **XML 断言说明**：HTTPRunner 对 XML 响应的 XPath 支持有限，采用 `contains` 匹配 XML 片段的方式替代 XPath，简单可靠。如需严格 XPath，可在 HTTPRunner 中配置自定义 Hook 解析 XML。

## 五、授信C系统设计（内置 Mock 版本）

### 为什么做内置 Mock 版本

直接用真实生产系统验收有风险，且环境不可控。在项目内实现一个 MODEL_001 的 mock 服务，可以：
- 形成完整的端到端闭环，真实跑通整个流程
- 作为 TestAgent 能力的验收标准环境
- 未来对接真实系统时只需修改 BASE_URL

### 授信C系统（内置 Mock）架构

```
TestAgent-PI（现有 Fastify 服务）
├── /api/chat/...          # 已有：对话接口
├── /api/conversation/...  # 已有：会话管理
└── /mock/model/score      # 新增：MODEL_001 被测接口  ← 新增
```

复用现有 PostgreSQL，新增 4 张业务表（与 SQL 预埋文件对应）。

### 新增数据库表（Prisma Schema）

```prisma
model UserInfo {
  userId    String  @id @map("user_id")
  userLevel Int     @map("user_level")
  @@map("user_info")
}

model AccountBalance {
  userId        String  @id @map("user_id")
  avg3mBalance  Decimal @map("avg_3m_balance")
  @@map("account_balance")
}

model CgsSocialSecurity {
  userId             String @id @map("user_id")
  socialSecurityFlag Int    @map("social_security_flag")
  @@map("cgs_social_security")
}

model SalarySummary {
  userId        String  @id @map("user_id")
  monthlySalary Decimal @map("monthly_salary")
  @@map("salary_summary")
}
```

### 被测接口规格

**入参**：POST `/mock/model/score`，`Content-Type: application/xml`

```xml
<?xml version="1.0" encoding="utf-8"?>
<REQUEST>
  <user_id>TC001</user_id>
  <model_id>MODEL_001</model_id>
  <channel>131</channel>
  <business_params>...</business_params>
</REQUEST>
```

**出参**：XML

```xml
<?xml version="1.0" encoding="utf-8"?>
<RESPONSE>
  <result_code>0000</result_code>
  <result_msg>success</result_msg>
  <admit_flag>1</admit_flag>
  <credit_limit>517500.00</credit_limit>
</RESPONSE>
```

### 业务逻辑实现（伪代码）

```typescript
// 1. 解析 XML，取出 user_id
// 2. 查询四张表
const { userLevel } = await db.userInfo.findById(userId)
const { avg3mBalance } = await db.accountBalance.findById(userId)
const { socialSecurityFlag } = await db.cgsSocialSecurity.findById(userId)
const { monthlySalary } = await db.salarySummary.findById(userId)

// 3. 准入判断
const admitted = monthlySalary > 10000 && socialSecurityFlag === 1

// 4. 系数取值
let coefficient = 0
if (admitted) {
  coefficient = avg3mBalance > 0 ? 2.3 : 0.5
}

// 5. 额度计算
let creditLimit = 0
if (admitted) {
  creditLimit = userLevel * avg3mBalance * monthlySalary * coefficient
  if (creditLimit < 0) creditLimit = 0
}

// 6. 返回 XML
return buildXmlResponse({ admitFlag: admitted ? 1 : 0, creditLimit })
```

### 完整闭环验收流程

```
①  TestAgent 对话：上传业务文档 → 生成 cases_summary.json
②  人工 Review JSON（确认规则理解正确）
③  TestAgent 对话：生成 sql/ + testcases/ 文件
④  执行 SQL → 数据写入 PostgreSQL（mock 服务的 DB）
⑤  httprunner run testcases/ --base-url http://localhost:8000
⑥  mock 服务接受请求 → 查 DB → 执行业务逻辑 → 返回 XML
⑦  HTTPRunner 断言校验 → 生成 HTML 报告
⑧  TestAgent 读取报告 → 输出 Markdown 摘要
```
