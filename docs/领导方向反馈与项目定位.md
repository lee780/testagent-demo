# 领导方向反馈与项目定位

> 记录时间：2026-03-24
> 背景：项目汇报后，领导对 TestPilot 方向的核心反馈

---

## 一、领导核心反馈（6条）

1. **不需要通用，要落地** — 就在授信系统中落地，从小的地方来，一个专项测试做好就足够了
2. **Agent 越小越稳定** — 不要做大而全的 Agent，越小越稳定，越大越不稳定
3. **定位是专业的授信工具** — 不是通用测试平台，是授信领域的专业测试工具
4. **从研发模式上来看** — 关注它如何融入研发流程，而不是独立产品
5. **落地解决项目组实际痛点** — 不是做大工具平台，而是解决项目组今天真正的问题
6. **团队协作** — 3人小组（1 Lead + 2 组员）共同推进，需要明确分工

---

## 二、方向解读：从"平台思维"切到"工具思维"

| 现状（平台思维） | 目标（工具思维） |
|--------------|--------------|
| 4种通用测试模式 | 聚焦授信场景的1-2种核心模式 |
| 通用 modelId 框架 | 硬编码授信专用配置 |
| 大而全的 Agent 工具集 | 只保留授信接口测试真正需要的工具 |
| 知识库 UI 管理模块 | 简化为固定配置文件 |
| 独立产品形态 | 嵌入研发流程的专项工具 |

---

## 三、下一步行动方向

### 3.1 聚焦授信系统痛点
在功能裁剪之前，先明确授信项目组测试上最头疼的 3 件事，从痛点倒推保留什么、砍掉什么。

### 3.2 Agent 瘦身原则
- 单一职责：每个 Agent 工具只做一件事
- 授信专用：系统提示词、工具参数、数据模板全部授信化
- 减少链路：降低工具调用层级，提升稳定性

### 3.3 融入研发模式
- 明确 TestPilot 在研发流程中的触发时机（提测前？联调后？回归时？）
- 与 CI/CD、需求管理、缺陷跟踪的衔接方式

---

## 四、团队分工建议

| 角色 | 负责方向 | 具体模块 |
|------|---------|---------|
| **Lead** | Agent 核心 + 架构决策 | `agent-core/`、系统提示词、工具设计、接口定义 |
| **组员 A** | 测试执行链路 | `mock/`、测试执行结果、报告模块、缺陷跟踪 |
| **组员 B** | 用户侧体验 | 前端 ChatView、用例库、知识库注入、文件下载 |

### 协作模式
```
main 分支（稳定）
├── feat/agent-xxx    ← Lead 分支
├── feat/report-xxx   ← 组员A 分支
└── feat/frontend-xxx ← 组员B 分支
```
- 各自在 feature branch 上开发
- PR → Lead review → 合入 main
- 接口先行：模块边界提前对齐，各自对着接口开发，不互相等待
